จดหมายจากคนที่เคยเสียเงินจริง

ถ้าคุณกำลังอ่านหนังสือนี้ คุณอาจกำลังคิดจะสร้าง trading system หรือกำลังนั่งดู backtest ที่ให้ผลตอบแทนปีละ 40% แล้วหัวใจเต้นแรง

หยุดก่อน — นั่นคือจุดที่อันตรายที่สุด

ผมเคยดู strategy มากกว่า 100 แบบที่ backtest "สวยมาก" แต่พังทันทีที่ใช้เงินจริง บางครั้งพังในสัปดาห์แรก บางครั้งพังในเดือนที่สาม แต่พังทุกครั้ง สิ่งที่ทำให้มันพังไม่ใช่โชคร้าย — มันคือ bias ที่ซ่อนอยู่ใน data และ model ที่เราไม่เห็น

เนื้อหาที่คุณกำลังจะอ่าน ไม่ใช่ตำราวิชาการ มันคือ แผนที่กับดักที่คนส่วนใหญ่ตกซ้ำแล้วซ้ำเล่า เขียนขึ้นเพื่อให้คุณรอดก่อน แล้วค่อยทำกำไรทีหลัง

บทที่ 1 · Chapter One
The Illusion of Backtesting
ภาพลวงตาของ Backtest
ทุก backtest คือการมองย้อนหลังด้วยสายตาที่รู้อนาคตแล้ว — มันไม่ใช่ความจริง มันคือนิยาย
The Core Idea · แก่นสาร

ทำไม Backtest 90% ถึงโกหกคุณ

เมื่อเราทำ backtest เราคิดว่าเราทดสอบว่า strategy ทำงานหรือไม่ในอดีต แต่ความจริงคือเราทำสิ่งที่แตกต่างกันมาก — เราหา pattern ในข้อมูลที่ผ่านไปแล้ว แล้วหลอกตัวเองว่ามันคือสัญญาณแห่งอนาคต

ปัญหาคือตลาดเงินมีสัญญาณรบกวน (noise) มหาศาล ถ้าเราดูข้อมูล 10 ปีและลอง parameter หลายร้อยแบบ เราจะพบ combination ที่ "ทำกำไรได้ในอดีต" เสมอ — ไม่ใช่เพราะมัน work แต่เพราะเราค้นหานานพอ เหมือนโยนเหรียญ 1,000 ครั้ง แล้วบอกว่า "เจอรูปแบบ" ว่าหัวออก 3 ครั้งติดกันก่อนก้อยเสมอ

Core Insight

Alpha จริง คือผลตอบแทนส่วนเกินที่เกิดจาก "ความได้เปรียบที่แท้จริง" ซึ่งคนอื่นยังไม่รู้หรือทำไม่ได้ ส่วน Alpha ปลอม คือผลตอบแทนที่ดูดีใน backtest แต่มาจากการค้นหา pattern ในสัญญาณรบกวน — มันจะหายไปทันทีที่ใช้เงินจริง

ถ้า strategy ของคุณมาจากการลอง parameter หลาย 100 แบบแล้วเลือกแบบที่ดีที่สุด — คุณกำลังสร้าง Alpha ปลอม

Alpha มาจากไหน — และอะไรที่ไม่ใช่ Alpha จริง

ความแตกต่างระหว่าง In-Sample และ Out-of-Sample

In-sample คือข้อมูลที่ใช้สร้างและปรับ strategy — เหมือนท่องจำข้อสอบ ส่วน Out-of-sample คือข้อมูลใหม่ที่โมเดลไม่เคยเห็น — เหมือนข้อสอบจริง คนส่วนใหญ่ปรับแต่ง strategy บน in-sample จนสมบูรณ์แบบ แล้วประหลาดใจเมื่อมันพังใน out-of-sample

Reflection Questions · คำถามสำหรับคิด
  1. Strategy ของคุณผ่าน out-of-sample test แล้วหรือยัง? ถ้ายัง นั่นไม่ใช่ strategy — มันคือสมมติฐาน
  2. คุณลอง parameter กี่แบบก่อนจะได้ผลลัพธ์ที่ดี? ยิ่งมาก ยิ่งต้องสงสัย
  3. ถ้าลบ 5 วันที่ทำกำไรดีที่สุดออกจาก backtest — performance ยังดีอยู่ไหม?

บทที่ 2 · Chapter Two
The Four Hidden Killers
4 Bias ที่ฆ่าพอร์ตโดยที่คุณไม่รู้ตัว
Bias เหล่านี้ไม่ได้เกิดจากความตั้งใจ — มันซ่อนอยู่ใน workflow ที่คุณใช้ทุกวัน
The Core Idea · แก่นสาร

ศัตรูที่มองไม่เห็น ใน Data ของคุณ

Bias ใน backtest ไม่ใช่สิ่งที่คุณตั้งใจทำ มันเป็นผลจากวิธีที่เราเก็บข้อมูล วิธีที่เราเลือกหุ้น และวิธีที่เราทดสอบ โดยไม่ระวัง เราสร้าง backtest ที่ดูดีแต่ผิดจากความเป็นจริงตั้งแต่แรก

นี่คือ 4 ตัวร้ายหลักที่ต้องรู้จัก:

Look-Ahead Bias
การมองอนาคตโดยไม่รู้ตัว
ใช้ข้อมูลที่ "ณ วันนั้นจริงๆ ยังไม่มี" ในการตัดสินใจ เช่น ตัวเลขงบการเงินที่ถูก revise หลังประกาศ หรือ closing price ที่ใช้ตัดสินใจ entry แต่ entry จริงทำได้แค่ราคา open วันถัดไป
ตัวอย่างในตลาดไทย
หุ้น SET บางตัวประกาศงบวันที่ 1 แต่ตัวเลขใน database ถูก revise วันที่ 15 — ถ้าใช้ตัวเลข revision ในการ backtest จะเห็นว่า "รู้ตัวเลขที่แม่นยำก่อนคนอื่น" ซึ่งในชีวิตจริงทำไม่ได้
Survivorship Bias
ความลำเอียงจากคนรอด
ข้อมูลใน database มีแต่บริษัทที่ "รอดมาถึงปัจจุบัน" บริษัทที่เจ๊งหรือถูก delist ออกไปแล้ว ทำให้ backtest ดูดีเกินจริง เหมือนสัมภาษณ์แค่คนที่รวยแล้วถามว่าทำอะไร โดยไม่สัมภาษณ์คนที่ล้มเหลวด้วย
ตัวอย่างในตลาดไทย
SET มีหุ้นถูก delist ไปหลายร้อยตัวในรอบ 10 ปี ถ้า database ของคุณไม่มีข้อมูลหุ้นเหล่านี้ performance จะดูดีกว่าความเป็นจริงอย่างมีนัยสำคัญ
Data Leakage
ข้อมูลรั่วไหลจากอนาคตสู่อดีต
เมื่อข้อมูลจาก "อนาคต" รั่วเข้าไปใน feature ที่ใช้ train model ทำให้ model "รู้" สิ่งที่ไม่ควรรู้ พบบ่อยมากใน machine learning — เช่น normalize data ด้วย mean ของ full dataset ก่อนแบ่ง train/test
กับดักพบบ่อย
การใช้ StandardScaler fit บน dataset ทั้งหมดก่อน split — ค่า mean และ std ของ test set รั่วเข้าไปใน training ทำให้ model ฉลาดเกินจริง
Overfitting
การจูนโมเดลเกินพอดี
โมเดลเรียนรู้ "สัญญาณรบกวน" ของข้อมูลในอดีตแทนที่จะเรียนรู้ "รูปแบบจริง" ยิ่งพารามิเตอร์มาก หรือยิ่งลอง combination มาก ยิ่งเสี่ยง overfitting — สัญญาณคือ in-sample performance สวยงามมากแต่ out-of-sample พัง
กฎง่ายๆ
ถ้า Sharpe ratio ใน in-sample สูงกว่า out-of-sample มากกว่า 2 เท่า — แสดงว่า overfit แน่นอน

Bias เหล่านี้แย่ขึ้นในตลาดไทยอย่างไร

ความเชื่อเดิม
ใช้ methodology เดียวกับตลาดอเมริกาได้เลย — ตลาดก็คือตลาด
↓

ตลาดไทย: liquidity ต่ำ, corporate action ถี่ (หุ้นปันผล, rights issue), ข้อมูลคุณภาพต่ำกว่า, historical data มักไม่ได้ปรับ adjusted price อย่างถูกต้อง — ทุกปัจจัยนี้ขยาย bias ให้รุนแรงขึ้น

Reflection Questions · คำถามสำหรับคิด
  1. Data ของคุณมีหุ้นที่ถูก delist ออกไปแล้วด้วยหรือไม่? ถ้าไม่มี ผล backtest ดีเกินจริงแน่นอน
  2. คุณ normalize หรือ scale data อย่างไร — มีโอกาส data leakage จาก test set หรือเปล่า?
  3. Strategy ของคุณมีกี่ parameter ที่ปรับแต่งได้? ยิ่งมาก ยิ่งต้องระวัง overfitting

บทที่ 3 · Chapter Three
The Thai Reality
ความจริงของตลาดไทย ที่ตำราฝรั่งไม่บอก
ตลาดไทยมีกฎพิเศษของตัวเอง — ถ้าไม่เข้าใจ backtest สวยแค่ไหนก็ไม่มีความหมาย
The Core Idea · แก่นสาร

SET ≠ S&P 500 — และมันสำคัญมากกว่าที่คิด

หนังสือ quant finance ส่วนใหญ่เขียนโดยคนที่ทำงานกับตลาดอเมริกา ซึ่งมี liquidity สูง data คุณภาพดี และนักลงทุนสถาบันครองตลาด แต่ตลาดไทย (SET) แตกต่างออกไปอย่างมีนัยสำคัญ ถ้าเอา methodology เดียวกันมาใช้โดยไม่ปรับ — ผลลัพธ์จะเป็นภัย

4 ความเป็นจริงของตลาดไทยที่ต้องเข้าใจ

⚠ ข้อจำกัดที่ 1 · Liquidity

หุ้น SET ส่วนใหญ่มี liquidity ต่ำมาก — หุ้นนอก SET50 มักมี average daily volume ต่ำกว่า 10 ล้านบาท ถ้าคุณ backtest ว่า "ซื้อหุ้น X 5% ของพอร์ต" ด้วยพอร์ต 10 ล้านดอลลาร์ ($10M) — นั่นหมายถึงซื้อ 500,000 ดอลลาร์ ซึ่งอาจเกิน 50% ของ daily volume หุ้นนั้นเลย ราคาจะขยับทันที่ที่คุณเข้า — นั่นคือ market impact ที่ backtest ไม่เคยคำนวณ

⚠ ข้อจำกัดที่ 2 · Retail Dominance

ตลาดไทยขับเคลื่อนโดย retail investor มากกว่า 60% ซึ่งหมายความว่าตลาดมีความผันผวนทางอารมณ์สูง momentum ทำงานได้แต่ก็กลับทิศเร็ว และราคาบางครั้งเคลื่อนโดยไม่มี fundamental รองรับ strategy ที่ใช้ได้ดีในตลาดสถาบัน อาจพังในตลาดที่ retail ครองได้

⚠ ข้อจำกัดที่ 3 · Regime Change บ่อย

ประเทศไทยมีการเปลี่ยนแปลงสภาพแวดล้อม (regime) บ่อยกว่าตลาดพัฒนาแล้ว ทั้งทางการเมือง (รัฐประหาร), เศรษฐกิจ (วิกฤต 97, COVID), และนโยบาย strategy ที่ทำงานดีในช่วง 5 ปีหนึ่ง อาจพังในอีก 5 ปีถัดมา เพราะสภาพแวดล้อมเปลี่ยนไปอย่างสิ้นเชิง

⚠ ข้อจำกัดที่ 4 · ข้อมูลคุณภาพต่ำ

ข้อมูล historical ของ SET มักมีปัญหา เช่น adjusted price ที่ปรับ corporate action ไม่ถูกต้อง, missing data ในช่วง trading halt, และ volume data ที่รวม odd lot จนบิดเบือน strategy ที่สร้างบน data ที่ไม่สะอาดจะมีผล backtest ที่เชื่อถือไม่ได้

กฎ Liquidity ที่ต้องใช้เสมอ

สำหรับพอร์ตขนาดใหญ่ในตลาดไทย ให้จำกัด position size ไว้ที่ไม่เกิน 5-10% ของ average daily volume (ADV) ของหุ้นนั้น ถ้าจะซื้อหุ้นที่มี ADV 5 ล้านบาท — position size สูงสุดของคุณคือ 250,000-500,000 บาท ไม่ใช่ล้าน

"A strategy that works on paper but can't be executed at scale is not a strategy — it's a fantasy."
Strategy ที่ทำงานในกระดาษแต่ execute ไม่ได้ด้วยเงินจริง — ไม่ใช่ strategy มันคือความฝัน
Reflection Questions · คำถามสำหรับคิด
  1. Backtest ของคุณคำนึงถึง market impact (การที่ราคาขยับเมื่อคุณซื้อ) แล้วหรือยัง?
  2. Strategy ทำงานได้ดีแค่ในช่วงเวลาใด? ถ้าแบ่ง test เป็น 3 ช่วงเวลา ทุกช่วงยังดีอยู่ไหม?
  3. ถ้า regime เปลี่ยน เช่น ดอกเบี้ยขึ้นฉับพลัน — strategy ของคุณจะเกิดอะไรขึ้น?

The Complexity Trap
กับดักความซับซ้อน — ยิ่งฉลาดยิ่งอันตราย
ตลาดไม่ให้รางวัลความฉลาด มันให้รางวัลกับคนที่ถูกต้องเสมอ ไม่ใช่คนที่ถูกต้องบางครั้งด้วยเหตุผลซับซ้อน
The Core Idea · แก่นสาร

Curve fitting คือการปรับ parameter จนโมเดลเข้ากับข้อมูลในอดีตอย่างสมบูรณ์แบบ — เหมือนตัด suit ให้พอดีกับร่างกายเฉพาะในวันนั้น พอน้ำหนักเปลี่ยนไม่กี่กิโล ชุดก็ใส่ไม่ได้แล้ว

A Trustworthy Backtest
Backtest ที่น่าเชื่อถือ — มาตรฐานขั้นต่ำที่ยอมรับไม่ได้
Backtest ที่ดีไม่ใช่แค่ตัวเลขสวย — มันคือกระบวนการที่โปร่งใส ทดสอบซ้ำได้ และออกแบบมาเพื่อ "หาจุดพัง" ไม่ใช่ "พิสูจน์ว่าดี"
The Core Idea · แก่นสาร

3 เสาหลักของ Backtest ที่เชื่อถือได้

ถ้าจะนับรวม backtest ว่าน่าเชื่อถือ มันต้องผ่านสามด่านนี้ ขาดด่านไหนไม่ได้เลย:

เสาที่ 1: PIT Data — ข้อมูล ณ เวลาจริง

PIT (Point-in-Time) Data คือข้อมูลที่บันทึกว่า "ณ วันนั้น เรารู้อะไรบ้าง" ไม่ใช่ข้อมูลที่ถูก revise ย้อนหลังในภายหลัง ตัวเลขงบการเงินหลายตัวมักถูก revise เช่น EPS ปรับจาก 1.50 บาทเป็น 1.65 บาทหลังตรวจสอบ — ถ้าใช้ตัวเลข 1.65 บาทในการ backtest ช่วงที่ประกาศ คุณกำลังโกงตัวเอง

✓ วิธีตรวจสอบ PIT Data

ถามผู้ให้บริการข้อมูลว่า "ตัวเลขนี้คือ as-reported หรือ restated?" ข้อมูล as-reported คือสิ่งที่เราต้องการ ส่วน restated คือข้อมูลที่ถูกปรับย้อนหลัง ซึ่งในชีวิตจริงเราไม่มีทางรู้ล่วงหน้าได้

เสาที่ 2: Walk-Forward Validation — การทดสอบแบบเดินหน้า

Walk-forward test คือการทดสอบที่จำลองวิธีการใช้งานจริง — Train model ด้วยข้อมูล 3 ปีแรก ทดสอบกับ 1 ปีถัดมา แล้ว retrain ด้วยข้อมูลที่อัพเดต ทดสอบกับอีก 1 ปีถัดมา ทำซ้ำจนครบ ผลลัพธ์คือ performance บนข้อมูลที่ "โมเดลไม่เคยเห็น" ทุก step — ใกล้เคียงความจริงมากที่สุด

Walk-Forward vs. Simple Train/Test Split

Simple split (80% train, 20% test) มีปัญหาคือ test set ถูกดูแค่ครั้งเดียว และนักพัฒนามักปรับ model จนทำงานดีกับ test set นั้นโดยรู้ตัวหรือไม่รู้ตัว นั่นก็คือ overfitting ต่อ test set

Walk-forward แก้ปัญหานี้เพราะโมเดลไม่เคยรู้ล่วงหน้าว่า "test set" ในแต่ละ step คืออะไร — มันต้องทำงานกับข้อมูลอนาคตจริงๆ

เสาที่ 3: Stress Testing — ทดสอบในสภาวะสุดโต่ง

ถาม strategy ว่า "เกิดอะไรขึ้นถ้า..." แล้วลองเปลี่ยนสภาวะตลาดให้รุนแรงขึ้น:

Stress Test ที่ควรทำ ✓
  • ทดสอบในช่วงวิกฤต (2008, COVID 2020, วิกฤต 97)
  • เพิ่ม slippage สมมติ 2-3 เท่า
  • ลด position size ได้สูงสุด 50% จาก liquidity
  • ทดสอบด้วย transaction cost สูง
  • ลอง parameter ±30% จากค่าที่เลือก
สิ่งที่ไม่ใช่ Stress Test ✗
  • แค่ดูว่า max drawdown คือเท่าไหร่
  • ทดสอบในตลาดขาขึ้นอย่างเดียว
  • ใช้ commission ที่ต่ำกว่าความจริง
  • ไม่ทดสอบกับ liquidity constraint
  • ข้ามช่วงเวลาที่ performance แย่
Reflection Questions · คำถามสำหรับคิด
  1. Data ที่คุณใช้คือ PIT หรือ restated? ถ้าไม่รู้ — นั่นคือความเสี่ยงที่ต้องตรวจสอบทันที
  2. คุณเคย stress test strategy กับช่วงวิกฤตจริงแล้วหรือยัง? ผลเป็นอย่างไร?
  3. ถ้า transaction cost สูงขึ้น 3 เท่า strategy ยังทำกำไรอยู่ไหม?

บทที่ 6 · Chapter Six
Robustness Over Performance
ความทนทาน สำคัญกว่าตัวเลขสวย
Strategy ที่ดีที่สุดไม่ใช่ strategy ที่ให้ผลตอบแทนสูงที่สุดในอดีต แต่คือ strategy ที่ "ยังทำงานได้" ในอนาคตที่เราไม่รู้
The Core Idea · แก่นสาร

Sharpe Ratio สูงไม่ใช่ข่าวดี

Sharpe Ratio คือตัวชี้วัดผลตอบแทนต่อความเสี่ยง — ยิ่งสูงยิ่งดี "ในทฤษฎี" แต่ใน backtest ถ้า Sharpe Ratio สูงผิดปกติ (เช่น 3.0+) นั่นเป็นสัญญาณเตือน ไม่ใช่เหตุผลที่จะดีใจ

เพราะ Sharpe สูงใน backtest มักมาจาก overfitting — โมเดลเรียนรู้ข้อมูลในอดีตจนแม่น แต่ข้อมูลอนาคตจะให้ผลต่างออกไป เหมือนนักเรียนที่ท่องจำข้อสอบเก่าได้ 100% แต่พอข้อสอบเปลี่ยน ทำได้แค่ 40%

Robustness คืออะไร — ในโลกจริง

Robustness (ความทนทาน) หมายถึง strategy ยังทำงานได้ดีพอเมื่อสภาวะเปลี่ยนไปจากที่เคย backtest ไว้ — ไม่ว่าจะเป็นสภาวะตลาด, parameter, หรือ data quality

Strategy ที่ไม่ Robust
ทำกำไรดีมากในช่วง 2019-2022 แต่พังหมดใน 2023 เมื่อดอกเบี้ยขึ้น — เพราะสร้างบน regime เดียวที่ไม่ยั่งยืน
↓
Strategy ที่ Robust
ทำกำไรได้สม่ำเสมอ 8-12% ต่อปีในทุกสภาวะ — ไม่สูงมาก แต่คงเส้นคงวา ผ่าน stress test ได้ทุกช่วง
↓
วิธีวัด Robustness
เปลี่ยน parameter ±30% แล้วดูว่า performance เปลี่ยนมากแค่ไหน — ถ้าเปลี่ยนนิดเดียวแล้วพัง นั่นคือ fragile ไม่ใช่ robust

Geometric Mean — วิธีวัดผลตอบแทนที่ตรงความจริง

Geometric Mean (ค่าเฉลี่ยเรขาคณิต) คือค่าเฉลี่ยที่คำนึงถึงการทบต้น ซึ่งต่างจากค่าเฉลี่ยปกติ (Arithmetic Mean) ตรงที่มันบอกผลตอบแทนจริงของเงินลงทุน

ทำไม Geometric Mean ถึงสำคัญ

สมมติปีแรกได้ +50% ปีที่สองเสีย -50% — ค่าเฉลี่ยปกติ = 0% (ดูเหมือนไม่ได้ไม่เสีย) แต่ Geometric Mean = -13.4% เพราะเงินจริงๆ หายไป: เริ่ม 100 บาท → ปีแรก 150 บาท → ปีที่สอง 75 บาท ติดลบ 25 บาท

Strategy ที่มี volatility (ความผันผวน) สูงจะมี Geometric Mean ต่ำกว่า Arithmetic Mean มาก — นั่นคือทำไม "ลด drawdown = เพิ่มผลตอบแทนจริงระยะยาว"

"The goal of risk management is not to minimize risk. It is to ensure the portfolio survives long enough for its edge to manifest."
เป้าหมายของการจัดการความเสี่ยงไม่ใช่การลดความเสี่ยงให้ต่ำที่สุด — แต่คือการทำให้พอร์ตรอดได้นานพอที่ความได้เปรียบของมันจะแสดงออกมา

Maximum Drawdown — ตัวชี้วัดที่สำคัญกว่า Sharpe

Drawdown คือการที่พอร์ตลดลงจากจุดสูงสุด เช่น พอร์ต 1 ล้านบาทลงไปเหลือ 700,000 บาท คือ drawdown 30% ถ้า Maximum Drawdown ใน backtest สูง (เช่น 40-50%) ถามตัวเองว่า "ถ้าเกิดขึ้นจริง คุณจะยังอยู่ใน strategy ได้ไหม? หรือจะตื่นตระหนกขายออก?"

17 สัญญาณว่า Strategy ของคุณ "ไม่น่าเชื่อถือ"

ก่อน deploy ให้ตรวจสอบสัญญาณเหล่านี้ทุกข้อ ถ้าพบแม้แต่ข้อเดียว — ต้องหยุดและตรวจสอบก่อน

🔴 Red Flags เรื่อง Data & Backtest

🚩
Sharpe Ratio ใน backtest สูงกว่า 2.5 — ยิ่งสูงยิ่งต้องสงสัย ในโลกจริงมีน้อยมากที่ทำได้แบบนี้ต่อเนื่อง
🚩
ไม่เคยทำ out-of-sample test — ถ้า backtest ทั้งหมดใช้ข้อมูล dataset เดียวกัน ผลไม่มีความหมาย
🚩
ไม่มีหุ้นที่ delist ออกไปใน data — survivorship bias ทำให้ผลดีเกินจริงมาก
🚩
ทำ trade จำนวนน้อยใน backtest (น้อยกว่า 200 ครั้ง) — sample น้อยเกินไปที่จะสรุปผลได้อย่างน่าเชื่อถือ
🚩
ไม่ได้คำนึงถึง transaction cost และ slippage — ในตลาดไทยที่ liquidity ต่ำ ค่าใช้จ่ายจริงสูงกว่าที่คิดมาก

🔴 Red Flags เรื่อง Model & Strategy

🚩
อธิบายไม่ได้ว่า "ทำไม" strategy ถึงควรทำงาน — ต้องมี economic rationale ที่ชัดเจน ไม่ใช่แค่ "เพราะ backtest บอกว่าดี"
🚩
เปลี่ยน parameter เล็กน้อยแล้ว performance เปลี่ยนมาก — strategy ที่ดีต้องทนต่อการ perturbation (การเปลี่ยนเล็กน้อย)
🚩
Strategy ทำงานดีเฉพาะในช่วงเวลาเดียว — ถ้าดีแค่ในตลาดขาขึ้น หรือดีแค่ในช่วงดอกเบี้ยต่ำ นั่นไม่ใช่ edge มันคือ beta ที่พรางตัว
🚩
โมเดลซับซ้อนมากกว่า 10 parameters — ทุก parameter เพิ่มโอกาส overfitting คุณต้องพิสูจน์ว่าแต่ละตัวจำเป็นจริง

🔴 Red Flags เรื่องตลาดไทย/Forex

🚩
Position size ใหญ่กว่า 10% ของ Average Daily Volume — คุณกำลังเป็นผู้ขับเคลื่อนราคาตัวเอง ไม่ใช่แค่ follower
🚩
ไม่ได้ใช้ adjusted price ที่ปรับ corporate action — ราคาที่ไม่ได้ปรับ rights issue/ปันผลจะทำให้ backtest บิดเบือน
🚩
ไม่ได้ทดสอบในช่วง regime change — ตลาดไทยผ่านวิกฤตมาหลายครั้ง ทดสอบทุกช่วงก่อนเสมอ
"If you think your strategy is too good to be true, it probably is. In 20 years of reviewing backtests, I've never seen a high Sharpe ratio survive live trading."
ถ้าคุณคิดว่า strategy ของคุณดีเกินจริง — มันก็อาจเป็นแบบนั้นจริงๆ ในประสบการณ์ 20 ปีที่ดู backtest มา ยังไม่เคยเห็น Sharpe Ratio สูงๆ ที่รอดได้จริงในตลาดจริง

Pre-Deploy Checklist — ทำครบก่อน deploy ทุกครั้ง

นี่คือ Non-negotiable (สิ่งที่ต้องทำเสมอ ไม่มีข้อยกเว้น) ก่อนนำ strategy ไปใช้เงินจริง:

1
Out-of-sample test ผ่านอย่างน้อย 2 ปี
Performance บนข้อมูลที่โมเดลไม่เคยเห็น ต้องไม่ต่ำกว่า 50% ของ in-sample performance
2
Walk-forward validation ผ่านอย่างน้อย 5 รอบ
ทุก fold ต้องทำกำไรได้ ไม่ใช่แค่ average ดี — 1-2 fold พังยังรับได้ แต่ 3+ ที่พัง ให้หยุดทันที
3
ผ่าน stress test ในช่วงวิกฤต
ต้องทดสอบกับ COVID crash 2020, วิกฤตต้มยำกุ้ง 1997 (ถ้ามีข้อมูล), และช่วงดอกเบี้ยขาขึ้น 2022
4
คำนวณ liquidity constraint จริง
Position size ต้องไม่เกิน 5-10% ของ Average Daily Volume ของแต่ละหุ้น และต้องรวม market impact ใน cost estimate
5
มี "Kill Switch" ที่กำหนดล่วงหน้า
กำหนดล่วงหน้าว่า "ถ้า drawdown ถึง X% หรือ consecutive losses ถึง Y ครั้ง จะหยุด strategy ทันที" โดยไม่ใช้ดุลยพินิจในตอนนั้น

  • ตรวจสอบว่า signal ยังเกิดขึ้นตามที่คาด
  • ดู drawdown ที่สะสมอยู่เสมอ
